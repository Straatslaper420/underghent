ROW → PIXEL

What one row in each tab becomes on screen.

Events row → UNDERGHENT marker popup + feed item

title date_start venue lat/lng genre artists support_acts price hour_end image_url ticket_url
Effective values are override ?? auto: latitude_override moves the marker, venue_override re-labels it, hide=TRUE removes the row from both sites, approved gates visibility. The row is matched to a venue circle by name/alias first, then nearest coordinates (~90 m) — matched events feed that circle's count badge and right-side feed.

fb_events row → cards on both sites

Read directly by UNDERGHENT (fetched as a second CSV, joined by gid) and by GHENTEVENT's event cards. Facebook going/interested counts also survive into Events rows when dedupe merges an FB duplicate into a higher-priority scrape.

Venues row → venue circle · feed panel · VENUE PORTALS entry

Edits here flow back via pull-venues into config/venues.json (full mirror), feeding normalize + geo enrichment — AND the frontend now fetches this tab directly at page load (v5): edit initials, coords or hide in the sheet and the map follows on the next reload, no code edit. Hover a portal entry = fly the map there; click = open its feed.

feesten row → GHENTEVENT listing card

Written by the standalone fiestenpipeline (full refresh, deduped by node ID since one event repeats across days). Same field names as fb_events, so the landing page parses both with the same code path.

GeoFail row → nothing (quarantine)

An event without coordinates can't be placed on the map, so it parks here. Type latitude_override + longitude_override on its row (or fix the venue in the registry) — the next run's pull-overrides picks them up and export promotes the row to Events (the GeoFail row is marked moved).
